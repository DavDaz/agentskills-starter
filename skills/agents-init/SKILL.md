## Purpose

You are responsible for restructuring an existing `AGENTS.md` into its optimized form.

The file was generated by `/init` (OpenCode or Claude Code). It has good raw content but
lacks structure, skills tables, and the 200-line budget discipline. Your job is to fix that.

See [template.md](template.md) for the canonical output structure with inline comments.

---

## THE 200-LINE RULE — NON-NEGOTIABLE

The AI agent runtime loads **only the first 200 lines** of `AGENTS.md` at session startup.
Everything after line 200 is loaded on demand. This is not a suggestion — it is a hard limit.

**Budget allocation (must fit within 200 lines):**

| Block | Target lines | Content |
|-------|-------------|---------|
| Header + How to Use | 1–8 | Project name, brief usage note |
| Skills Reference | 9–55 | List or table of available skills |
| Auto-invoke Skills | 56–130 | Action → Skill mapping table |
| Critical Rules | 131–200 | ALWAYS/NEVER non-negotiable rules |

**Content that goes AFTER line 200:**
Tech Stack, Project Structure, Commands, QA Checklist, Naming Conventions.

If after restructuring the high-priority content exceeds 200 lines, trim verbose descriptions —
not sections. Every section listed above must appear.

---

## Step 1 — Read existing AGENTS.md

Read `AGENTS.md` and extract the following into internal working variables:

- **project_name**: from the first `#` heading
- **tech_stack**: any mention of languages, frameworks, versions
- **commands**: any bash blocks (build, test, lint, run)
- **project_structure**: any directory tree or component table
- **critical_rules**: any ALWAYS/NEVER/MUST/NEVER rules or non-negotiables
- **naming_conventions**: any naming tables or patterns
- **qa_checklist**: any checklist items
- **other_content**: anything else not covered above

Preserve all content — you will reorganize it, not discard it.

---

## Step 2 — Discover skills

Scan for `SKILL.md` files using this glob: `skills/*/SKILL.md`

For each skill found:
1. Read the YAML frontmatter — extract `name` and `description`
2. If no frontmatter, use the directory name as `name` and the first paragraph as `description`
3. From the `description`, extract **trigger phrases** — look for:
   - Patterns: `"Use when [phrase]"`, `"Trigger: [phrase]"`, `"Triggers: [phrase]"`
   - Action verb phrases: `Creating X`, `Modifying Y`, `Writing Z`, `Adding X`
   - Explicit "when" clauses

Build a list:
```
skills_found = [
  { name: "skill-name", description: "...", path: "skills/skill-name/SKILL.md", triggers: ["..."] },
  ...
]
```

If `skills/` is empty or does not exist, proceed with `skills_found = []` and use placeholder tables.

---

## Step 3 — Build Skills Reference section

**If skills_found has ≤ 8 skills**, use the compact list format (saves vertical space):

```markdown
> **Skills Reference** — load these on demand for detailed patterns:
> - [`skill-name`](skills/skill-name/SKILL.md) — One-line description, max 80 chars
> - [`skill-name-2`](skills/skill-name-2/SKILL.md) — One-line description
```

**If skills_found has > 8 skills**, use a table:

```markdown
## Available Skills

| Skill | Description | File |
|-------|-------------|------|
| `skill-name` | One-line description | [SKILL.md](skills/skill-name/SKILL.md) |
```

**If skills_found is empty**, use this placeholder:

```markdown
> **Skills Reference** — no skills found in `skills/`.
> Add skills at `skills/<name>/SKILL.md` and run `/agents-init` to sync.
```

---

## Step 4 — Build Auto-invoke table

Using `trigger phrases` extracted in Step 2, build:

```markdown
### Auto-invoke Skills

When performing these actions, ALWAYS load the corresponding skill FIRST:

| Action | Skill |
|--------|-------|
| Trigger phrase | `skill-name` |
```

Rules:
- One skill can have multiple rows (one per trigger phrase)
- Sort rows alphabetically by Action column
- Capitalize the first word of each action phrase
- If a skill has no extractable triggers, add: `Working with <skill-name topic>` | `skill-name`
- If `skills_found` is empty, add a placeholder row:
  `| _No skills registered yet — run /agents-init after adding skills_ | — |`

---

## Step 5 — Assemble AGENTS.md

Compose the file in this exact order. Track line count as you build.

```
[L1–8]     Header block (see template.md for exact format)
[L9–~55]   Skills Reference (Step 3 output)
[L~56–~130] Auto-invoke table (Step 4 output)
[L~131–200] Critical Rules — from extracted critical_rules
             If none found: use the placeholder from template.md
[L201+]    Tech Stack        ← from extracted tech_stack
           Project Structure ← from extracted project_structure
           Commands          ← from extracted commands
           QA Checklist      ← from extracted qa_checklist
           Naming Conventions← from extracted naming_conventions
           Other content     ← anything remaining
```

**Content rules:**
- If a section has no extracted content, use the placeholder from `template.md` — never omit a section
- Do not invent content — use only what was in the original AGENTS.md or the placeholders
- The divider comment `<!-- ══════ 200-LINE BUDGET ══════ -->` must appear around line 200

---

## Step 6 — Validate before writing

Before calling Write, verify:

1. **Line count**: count lines in the assembled content
2. **Budget check**: confirm Skills Reference + Auto-invoke + Critical Rules fit within lines 1–200
3. **Completeness**: confirm no section from the original file was dropped
4. **Skill count**: confirm all skills from `skills_found` appear in both tables

If budget is exceeded (high-priority content > 200 lines):
- Trim description text in the Skills Reference to 60 chars max
- Trim verbose ALWAYS/NEVER rule text (keep the rule, cut the explanation)
- Do NOT remove any section

---

## Step 7 — Write and report

Write the complete restructured content to `AGENTS.md`.

Then report to the user:

```
✅ AGENTS.md restructured

Lines total:         [N]
Lines 1-200 (startup budget used): [N] / 200
Skills discovered:   [N] — [name1, name2, ...]
Auto-invoke rows:    [N]
Content preserved:   [list of sections found and carried over]
Content placeholded: [list of sections not found in original — placeholders inserted]
```
